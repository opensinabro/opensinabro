# 나무위키 파리티 검증

나무위키 본체(the seed)는 비공개이고 namu.wiki는 크롤링을 차단한다(403). 그래서 "우리
구현이 실제 나무위키와 의미론상 일치하는가"를 직접 확인할 1차 근거가 없다. 이 문서는
그 근거를 얻는 경로와, 얻은 차이를 0으로 수렴시키는 작업법을 남긴다.

문법 사실 자체는 여기가 아니라 [나무마크 문법 스펙](../../docs/spec/namumark.md)과
`fixtures/corpus/`가 정본이다. 근거 우선순위 4단계도 스펙 문서에 있다.

## openNAMU를 단독 근거로 쓰지 않는 이유

실례가 있다. openNAMU에 있는 `{{{#!if}}}`를 "openNAMU 자체 확장"으로 판단한 적이 있는데,
실제로는 **the seed도 지원한다**(`틀:상위 문서`가 쓴다). openNAMU에 있다고 확장인 것도,
없다고 없는 것도 아니다. 판단은 the seed 실동작과 문법 도움말로 한다.

## 도구 셋

세 도구가 서로 다른 사각지대를 본다. **AI 없이 돌아가는 것이 요구사항이다** — 매번 대조를
사람이나 모델이 하면 비용이 한계를 정한다.

| 도구 | 잡는 것 | 네트워크 |
|---|---|---|
| `parity compare` | the seed 렌더와 우리 렌더의 구조 차이 | 코퍼스 fetch 시에만 |
| `parity report` | 우리가 **조용히 버린** 매크로·옵션 | 불필요 |
| `residue_scan` | 우리가 **텍스트로 흘린** 마크업 | 불필요 |

```sh
python3 ../fetch-parity-corpus.py          # 코퍼스 수집 (target/parity-corpus/, 커밋 안 함)
cargo run -p parity -- compare             # the seed ↔ 우리 렌더 대조
cargo run -p parity -- report              # 미해결 매크로·옵션 빈도
cargo run -p namumark-parser --example residue_scan -- <문서디렉토리>/*.namu
```

## 하네스가 아니면 못 잡는 부류

`residue`도 골든도 못 보는데 `parity compare`만 잡은 사례다 — 하네스를 만든 이유가 여기 있다.

- **표 전체(`table`) 스코프 속성 누락** — `tablewidth`·`tablebgcolor` 등이 화면에서 통째로
  사라졌다. 그런데 마커는 정상 소비되어 `residue`는 0이고, 골든은 우리 출력끼리만 비교하니
  무변경으로 통과했다. the seed 렌더 대조만이 잡았다.
- **`#!wiki` style 화이트리스트** — the seed는 `image-rendering`·무효 값(`display: 5ine`)·
  함수 안 함수가 든 값을 걸러 낸다. 우리는 무필터로 통과시켜 도움말 서술과도 어긋났다.
  렌더로 통과 속성을 목록화해 `style.rs`(`SupportedStyle`)로 재현했다.

## 작업법

**건수는 진척도의 지표가 아니다.** 어긋난 지점 하나가 뒤따르는 조각들의 정렬을 통째로
밀어내 건수를 부풀린다. `compare`는 두 가지를 함께 보여준다.

- **케이스마다 맨 앞 차이 3건** — 여기가 근원이다. 뒤따르는 것은 대개 밀려서 나온 결과다.
  구조가 어긋나는 유형(태그가 다른 경우)은 이쪽을 봐야 원인이 잡힌다.
- **차이 유형 분포** — 조각 하나짜리 차이(속성 값 등)는 정렬을 밀지 않으므로 분포 상위에
  그대로 쌓인다. 이쪽은 개수가 곧 영향도다.

순환은 이렇다: `compare` 실행 → 맨 앞 차이로 근원 파악(또는 유형 상위로 영향도 파악) →
the seed 렌더 HTML에서 실제 동작 확정 → 수정 → 재측정.

문법 도움말·심화 두 문서를 이 방법으로 13,571건에서 0으로 수렴시켰다. 유효했던 것 둘:

- **문서 순서대로 앞에서부터 일치**시킨다. 건수를 줄이는 게 목표가 아니다 — 맨 앞 차이를
  고치고 다시 재면 그다음 차이가 드러난다.
- **위키 UI는 무시가 아니라 제거**: `known-differences.txt`는 차이를 통과시킬 뿐 정렬은
  어긋난 채라 이후가 전부 밀린다. the seed에만 있는 편집 링크(`wiki-edit-section`)·
  분류 바(`wiki-categories`)처럼 마크업 산출물이 아닌 것은 정규화가 서브트리째 들어내야
  양쪽이 다시 맞물린다(`is_dropped_subtree`).

정규화가 흡수하는 표현 차이(클래스 어휘·색상 표기·style 순서)는 `normalize.rs`가 설명한다.
다크 색상(`data-dark-style` ↔ CSS 변수)은 비교에서 제외한다 — 표 스코프 다크 색은
[구현 현황](../../docs/spec/implementation-status.md)에 남은 과제로 올라 있다.

## 각 도구의 한계

**`residue_scan`은 후보 발굴용이고, 확정은 렌더 대조로 한다.**

- **거짓 양성**: 실제 문서에는 `'''`나 `[[`를 닫지 않은 오작성이 흔하다. 이건 나무위키에서도
  깨져 보이므로 우리 버그가 아니다. 문법 도움말 문서는 문법을 *설명*하느라 예제를 리터럴로
  싣기 때문에 특히 신호 대 잡음비가 나쁘다.
- **거짓 음성**: 마커가 소비되기만 하면 residue는 0이다. 파라미터를 잘못 해석하거나 조용히
  무시하는 차이는 잡지 못한다(예: 이미지의 미지원 옵션은 폐기되어 화면에도 residue에도
  안 남는다). 이 사각지대를 `parity report`가 덮는다.

**`parity report`**는 매크로 이름 판정이 `char::is_alphanumeric`이라 `[V]`, `[Chorus]` 같은
평범한 대괄호 텍스트가 매크로로 잡힌다. **빈도가 높은 이름**에 주목한다 — 실제 문서가 널리
쓰는 이름이면 the seed가 지원할 가능성이 크다.

## 근거 등급 `렌더확정`

`fixtures/corpus/`의 등급 체계에서 `렌더확정`은 "the seed 렌더 HTML에서 대응 출력을 실제로
확인함"을 뜻한다. 코퍼스 케이스는 아직 케이스별로 그 확인을 하지 않았으므로 이 등급을 쓰지
않는다 — `도움말예제`는 확인이 *가능하다*는 뜻이지 *했다*는 뜻이 아니다. 등급 체계 전반은
`fixtures/corpus/README.md`에 있다.

**임의의 마크업은 실측할 수 없다.** 알파위키에 지어낸 마크업을 넣어 the seed로 렌더시키는
읽기 전용 경로는 없다 — `/preview`는 POST 전용이고 쓰기 경로는 쓰지 않는다.
