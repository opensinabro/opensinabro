# 위키 서버 아키텍처

코드에서 읽어낼 수 없는 판단 기준만 남긴다. 크레이트가 무엇을 하는지는 각
`lib.rs`의 문서주석이, 테이블 정의는 `crates/wiki-server/migrations/`가,
API 경로는 `wiki-server`의 `router()`가 정본이다.

목표는 **자체 호스팅 1순위**다 — `docker compose up` 한 번으로 곧장 돌아가야 한다.
아래 선택 상당수가 이 제약에서 나온다.

## 크레이트 분할: 계층이 아니라 데이터 소유권

각 크레이트가 자기 부분영역의 타입·규칙·저장·질의를 **전부** 소유한다.

초안 두 번은 기술 계층(순수 규칙 / 영속화 / 유스케이스 조율 / HTTP)으로 잘랐는데,
그 기준은 MECE가 아니었다. 기능 하나(예: 토론)가 네 크레이트에 조각나 얹히고,
"유스케이스 조율"은 무엇이든 담기는 잡동사니 계층이 된다. 나무마크 계열이 MECE인
이유는 계층이어서가 아니라 **크레이트마다 소유하는 산출물(CST·AST·IR·마크업)이 다르기
때문**이다. 위키 서버에서 그에 대응하는 소유물은 부분영역의 데이터다.

따르는 규칙:

- **의존은 참조 방향이다.** 리비전이 actor를 참조하므로 document → account, 토론이 문서에
  달리므로 discussion → document. **account는 아무것도 참조하지 않는 뿌리**이고, 이
  불변식이 순환을 막는다. 사용자 기능이라도 문서를 FK로 참조하면 document 쪽이 소유해야
  한다(문서 구독 `star`가 account가 아닌 document 소유인 이유). 알림처럼 참조가 느슨한
  것은 FK 대신 제목을 payload에 담아 방향을 만들지 않는다.
- **부분영역 간 정합은 최상위에서.** "편집 = 권한 판정 → 리비전 저장 → 색인 갱신"처럼
  크레이트를 가로지르는 흐름은 `wiki-server`(와 덤프 임포터)가 조립한다. 전용 조율
  크레이트는 두지 않는다 — 그것이 잡동사니가 되는 자리다.
- **두지 않는 것**: repository trait 간접층(sqlx 직접, 테스트는 SQLite in-memory),
  공유 "domain 타입" 크레이트(참조 타입은 소유 크레이트가 공개한다).

**마이그레이션만은 크레이트별로 흩지 않는다.** 크레이트가 교차 FK(document → account 등)를
가지므로 적용에 단일 순서가 필요하다. 그래서 논리적 소유(크레이트)와 물리적 적용
순서(서버 바이너리의 단일 `migrations/`)를 분리한다 — 파일명 접두어가 소유 크레이트를
가리키지만 번호는 전역이다.

## 기술 선택

| 영역 | 선택 | 근거 |
|---|---|---|
| DB | PostgreSQL (sqlx) | 위키의 쓰기 패턴(동시 편집·리비전 채번)에 맞는 동시성. 부분 유니크 인덱스·JSONB·TIMESTAMPTZ를 데이터 모델이 전제한다 |
| 검색 | tantivy (한국어는 문자 n-gram) | 임베디드 전문 검색이라 외부 검색엔진 의존이 없다. 형태소 사전은 정확도가 낫지만 외부 사전 파일이 생겨 "받아서 곧장 실행"과 어긋나므로 선택적 확장으로 미룬다. the seed의 Elasticsearch 분리는 규모가 커질 때의 선택지로만 |
| HTTP | axum (tokio·tower) | tower 미들웨어로 세션·요청 제한을 계층화 |
| 프론트엔드 | Next.js (App Router, RSC) | 본문 HTML은 `backend-namuwiki`가 이미 방출하므로 셸만 담당한다 |

## URL 설계

**원칙: 일반적인 웹페이지 경험을 따른다. the seed URL 호환은 목표가 아니다.**

코드 곳곳에 "the seed를 따른다"와 "따르지 않는다"가 섞여 있는데, 그 판단 기준이 이것이다.
동작의 충실도는 따르고, 표기 관례는 웹 관례를 따른다.

- 소문자 kebab-case. the seed의 PascalCase 특수 페이지(`/RecentChanges`)는
  `/recent-changes`로, 별도 `/Go` 경로는 두지 않고 검색이 완전 일치 시 리다이렉트한다.
- 표준 HTTP 의미론: 조회는 GET, 권한 거부 403, 없는 문서 404(빈 문서 안내 화면으로),
  리다이렉트 문서는 302 + `?from=` 표시.
- 제목은 표준 퍼센트 인코딩. 렌더러의 링크 인코딩 규칙과 **같은 함수를 공유**해 문서 안
  링크와 주소창 표기가 일치하게 한다.

**문서 동작은 동사 접두사**(`/w/`·`/edit/`·`/history/`)를 쓴다. 접미사(`/w/<제목>/history`)가
더 흔한 관례지만, 나무위키 제목은 `/`를 포함할 수 있어(하위 문서 `상위/하위`) 접미사가
제목의 일부인지 동작인지 모호해진다. `/`나 `..`처럼 경로로 표현 못 하는 제목은 `?doc=`
fallback.

axum이 직접 처리하는 것은 `/api/` 아래의 JSON API와 `/file/`(바이너리 서빙)뿐이고,
나머지는 화이트리스트가 아니라 **fallback으로** Next.js에 넘긴다 — 화면이 늘어도 라우팅을
고치지 않는다. 프론트엔드가 axum 뒤에 서므로 오리진이 하나이고 CORS·SameSite 문제가 없다.
대신 `X-Forwarded-For`로 클라이언트 IP를 전파해야 한다. IP 사용자 편집·`ip:CIDR` 조건·
차단이 전부 이 값에 걸려 있어, 루프백으로 뭉개지면 권한 판정이 통째로 무너진다.

## 보안 표준

- 세션 쿠키는 `HttpOnly`·`SameSite=Lax`·(HTTPS에서)`Secure`. 쿠키에 담는 것은 내부 id가
  아니라 외부 식별자(UUID)이고, 로그인 시 세션 식별자를 새로 발급해 세션 고정을 막는다.
- 상태를 바꾸는 모든 요청에 CSRF 토큰(동기화 토큰 패턴). 브라우저 JS는 `HttpOnly` 쿠키를
  읽지 못하므로 토큰은 서버가 본문으로 따로 내준다.
- 비밀번호는 argon2. 로그인 실패는 이름과 비밀번호 중 어느 쪽이 틀렸는지 구분해 알리지
  않는다 — 계정 존재 여부가 새지 않게.
- 모르는 ACL 조건은 `AclCondition::Unsupported`로 두어 **매치되지 않게** 한다. 모르는
  조건을 허용으로 읽어 권한이 새는 것을 막는 fail-closed 규칙이다.
- 위키 입력이 원시로 나가는 통로는 렌더러가 이미 화이트리스트 처리한다(`#!html`,
  `#!wiki` style).

## 프론트엔드 상태 계층

서버 상태는 RSC가 서버에서 읽어 내려보내고, 갱신이 필요한 상호작용(검색 자동완성·토론
폴링·편집 미리보기)만 TanStack Query가 맡는다. Zustand는 편집기 초안처럼 서버에 없는
로컬 상태 전용이다. **캐시를 겹쳐 두지 않는다** — 서버가 매 요청 렌더하고, Next 계층은
문서 저장 시 재검증 신호를 받는 얇은 층으로 둔다.

셸이 쓰는 것은 `/api/session` 하나로 모은다(위키 이름·로그인 상태·안 읽은 알림 수·CSRF
토큰). 화면마다 세션을 따로 물으면 일부 화면만 로그인 상태를 싣지 않아 그 화면의 폼만
403을 내는 결함이 생긴다 — 실제로 겪었다.

API 응답 타입은 Rust에서 TypeScript 정의를 **생성**한다. 양쪽에 손으로 적으면 반드시
어긋난다.

## 데이터 모델 원칙

- **문서의 항등은 id, 제목은 속성이다.** 이동·맞바꾸기는 제목 컬럼 갱신일 뿐이고
  역사·토론·ACL은 id를 따라간다.
- **리비전은 append-only 전문(全文) 저장.** 원문 바이트 그대로 보존한다(파서의 무손실
  원칙). 삭제·이동·되돌리기도 리비전의 한 종류다 — 최근 변경·복원·감사가 전부 리비전
  조회로 환원된다. diff·blame은 조회 시 계산한다.
- **내부 식별자와 외부 식별자는 완전히 분리한다.** 정수 PK와 외래키는 DB 내부 전용이고
  URL·HTML·API 어디에도 나가지 않는다. 생성 순서·개체 수가 새지 않게 하려는 것이고,
  외부 노출이 필요하면 UUID v4나 자연 키(문서 제목, 사용자명)로만 참조한다.
- **파생 자료는 원본에서 재구성 가능해야 한다.** 역링크·검색 색인은 전부 리비전에서 다시
  만들 수 있다 — 스키마 변경·장애 복구 시 재구축이 탈출구다.

나머지 원칙(열거 테이블, actor 통일, 이력으로 쌓기)은 해당 마이그레이션 파일의 헤더
주석에 근거와 함께 있다.

## 접지 않고 남겨 둔 것

결론만 코드에 남아 이유를 잃기 쉬운 것들이다.

- **렌더 결과는 저장하지 않는다.** 렌더 캐시를 두었다가 걷어냈다(마이그레이션 0010).
  무효화 근거로 쓸 수 있는 것이 1단계 역링크뿐인데, include는 연쇄하고(A가 B를, B가 C를
  include하면 C의 변경이 A까지 닿는다) 리다이렉트 대상 판정도 렌더 결과 바깥에 있어,
  어떤 무효화 규칙을 세워도 stale이 남았다. 보기 요청은 매번 렌더한다.
- **`WikiContext`를 async trait으로 바꾸지 않는다.** `&dyn WikiContext`로 쓰므로 dyn-safe
  async가 필요하고, 그러면 조회마다 `Box<dyn Future>` 할당이 생겨 렌더 경로의 할당
  기준(`allocation_probe`)과 정면으로 충돌한다. 참조 수집 전용 API를 따로 두는 안도
  resolve의 순회를 복제하게 되어 수집과 렌더 의미론이 어긋날 위험을 되살린다. 대신
  렌더러가 무엇을 묻는지 기록하는 컨텍스트로 한 번 렌더해 필요한 제목을 모으고, 답을 채운
  스냅샷으로 다시 렌더한다.
- **캡차는 넣지 않는다.** reCAPTCHA 같은 외부 서비스는 자체 호스팅 1순위와 어긋나고,
  자체 구현은 효과가 미미하다. 스팸 대응은 지금 있는 수단(aclgroup 차단·요청 제한·
  편집요청 유도)으로 두고, 필요해지면 설정으로 붙일 자리로 남긴다.
- **본문과 셸의 경계를 코드 경계로 쓴다.** `backend-namuwiki`가 본문(`wiki-paragraph`
  어휘)을, 프론트엔드가 셸을 그린다. 파리티 대조에서 확인했듯 the seed도 본문 클래스는
  공개 어휘이고 셸은 스킨 소관이다. 프론트엔드는 본문 HTML을 그대로 주입하고 스타일을
  덮지 않는다 — 이 격리가 깨지면 파리티 스냅샷은 통과하는데 화면만 어긋난다.

## 비목표

- the seed 내부 구현 모방 — 근거는 관찰 가능한 동작뿐이다.
- 다크 테마, 다중 위키 호스팅, 외부 검색엔진 필수 의존.
- openNAMU 자체 확장(게시판, 도전과제)의 재현.
