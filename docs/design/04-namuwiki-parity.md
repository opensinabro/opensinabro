# 설계: 나무위키 파리티 검증

상태: 검증 체계 확립 (2026-07)

## 문제

나무위키 본체(the seed)는 비공개라 나무마크의 공식 스펙이 없다. namu.wiki는 크롤링을
차단한다(403). 따라서 "우리 구현이 실제 나무위키와 의미론상 일치하는가"를 직접 확인할
1차 근거가 없다.

## 근거 우선순위

같은 문법이라도 근거마다 신뢰도가 다르다. 판단이 갈리면 위쪽을 따른다.

| 순위 | 근거 | 성격 | 한계 |
|---|---|---|---|
| 1 | **the seed 엔진의 실제 렌더링 결과** | 확정 | 나무위키 본체는 접근 불가 → 아래 대체 경로 |
| 2 | **나무위키 문법 도움말 원문** | 스펙 서술 | 서술과 실제 동작이 어긋날 수 있음 |
| 3 | 실제 문서 덤프의 사용 양상 | 빈도·실재성 | 문서 원문 자체의 오작성이 섞임 |
| 4 | openNAMU 렌더러 소스 | 참고 | 나무위키에 없는 자체 확장이 섞여 있어 단독 근거로 쓰지 않는다 |

4순위를 낮게 두는 이유의 실례: openNAMU에 있는 `{{{#!if}}}`를 "openNAMU 확장"으로 판단한
적이 있는데, 실제로는 **the seed도 지원한다**(`틀:상위 문서`가 쓴다). openNAMU에 있다고
확장인 것도, 없다고 없는 것도 아니다. 판단은 1·2순위 근거로 한다.

### the seed 렌더링 결과를 얻는 경로

**알파위키**(`www.alphawiki.org`)가 나무위키와 **같은 the seed 엔진**을 쓰면서 원문과
렌더링 결과를 모두 공개한다. 나무위키 대신 여기를 1차 근거로 삼는다.

- 렌더링 결과: `GET /w/<문서명>` — HTML 그대로
- 원문 마크업: `GET /raw/<문서명>` — 본문은 `window.INITIAL_STATE`에 base64(zlib(protobuf))로 들어 있다.
  textarea는 비어 있고 클라이언트가 채우므로 HTML만 봐서는 안 되고 이 값을 풀어야 한다.

알파위키의 `알파위키:문법 도움말`, `알파위키:문법 도움말/심화`는 문서 말미의
`틀:문서 가져옴/나무위키`가 밝히듯 **나무위키의 문법 도움말을 그대로 가져온 문서**다.
따라서 이 두 문서는 순위 2(스펙 서술)인 동시에, 그 렌더링 결과가 순위 1(실제 동작)이다.

문법 도움말은 예제를 "입력 / 출력" 표로 나란히 싣는다. **원문에서 예제 마크업을 찾고
렌더 HTML에서 대응 출력을 찾으면 the seed의 실제 동작이 확정된다.** 이것이 이 저장소가
쓰는 대조 기법이다.

읽기(GET)만 사용한다. 편집·미리보기 등 쓰기 경로는 쓰지 않는다.

### 실제 문서 덤프

HuggingFace `heegyu/namuwiki` 덤프(2022-03-01, 867,024건)의 datasets-server rows API로
무작위 표본을 얻어 빈도를 잰다. 발견한 차이가 실제로 얼마나 흔한지 판단하는 데 쓴다.
문서 본문은 CC BY-NC-SA 2.0 KR이므로 `fixtures/`의 라이선스 분리 규정을 따른다.

## 도구

세 도구가 서로 다른 사각지대를 본다. **AI 없이 돌아가는 것이 요구사항이다** — 매번 대조를
사람이나 모델이 하면 비용이 한계를 정한다.

| 도구 | 잡는 것 | 네트워크 |
|---|---|---|
| `parity compare` | the seed 렌더와 우리 렌더의 구조 차이 | 코퍼스 fetch 시에만 |
| `parity report` | 우리가 **조용히 버린** 매크로·옵션 | 불필요 |
| `residue_scan` | 우리가 **텍스트로 흘린** 마크업 | 불필요 |

### 파리티 대조 (`parity compare`)

```
python3 tools/fetch-parity-corpus.py     # 코퍼스 수집 (target/parity-corpus/, 커밋 안 함)
cargo run -p parity -- compare           # the seed ↔ 우리 렌더 대조
```

the seed 렌더 결과와 우리 백엔드 출력을 **구조 지문**으로 정규화해 대조한다. 두 출력은
골격(태그·중첩·style 시맨틱)이 같고 표현만 다르므로, 표현 차이를 걷어내면 남는 diff가
곧 의미론 차이다.

| 표현 차이 | the seed | 우리 | 정규화 |
|---|---|---|---|
| 클래스 어휘 | `class='t-m-VAGP'` | `class="wiki-table"` | 양쪽 제거 |
| 색상 표기 | `rgb(171,205,239)` | `#abcdef` | 6자리 hex로 통일 |
| style 순서·공백 | `width:30%; text-align:center` | `text-align: center; width: 30%` | 파싱 후 이름순 정렬 |
| 다크 색상 | `data-dark-style` | CSS 변수 + `.dark` | 비교 제외 (별도 과제) |

`tools/parity/known-differences.txt`에 등록한 패턴에 걸리는 차이는 통과시키고, **새 차이만**
보고하며 있으면 종료 코드 1이다. 의도된 차이를 등록할 때는 근거를 주석으로 남긴다.

코퍼스 수집(`fetch-parity-corpus.py`)은 `[include(틀:...)]`가 참조하는 틀 문서까지 재귀로
받는다. 파리티 도구가 include를 확장하려면 틀 원문이 필요하기 때문이다. 문서 본문은
CC BY-NC-SA 2.0 KR이라 캐시는 커밋하지 않는다(`target/` 아래).

**건수는 진척도의 지표가 아니다.** 어긋난 지점 하나가 뒤따르는 조각들의 정렬을 통째로
밀어내 건수를 부풀린다. `compare`는 두 가지를 함께 보여준다.

- **케이스마다 맨 앞 차이 3건** — 여기가 근원이다. 뒤따르는 것은 대개 밀려서 나온 결과다.
  구조가 어긋나는 유형(태그가 다른 경우)은 이쪽을 봐야 원인이 잡힌다.
- **차이 유형 분포** — 조각 하나짜리 차이(속성 값 등)는 정렬을 밀지 않으므로 분포 상위에
  그대로 쌓인다. 이쪽은 개수가 곧 영향도다.

작업 순환은 이렇다: `compare` 실행 → 맨 앞 차이로 근원 파악(또는 유형 상위로 영향도 파악)
→ the seed 렌더 HTML에서 실제 동작 확정 → 수정 → 재측정. **여기에 AI가 필요 없다.**

**현재 상태: 문법 도움말·심화 두 문서 모두 파리티 0.** 13,571건에서 시작해 앞에서부터
차례로 일치시켜 0으로 수렴했다(알려진 차이 도움말 2·심화 16은 전부 hljs 구문강조 등
정규화가 흡수하는 몫). 코퍼스를 넓히거나(다른 문서 수집) CI 게이트로 전환하는 것이 다음
순서다.

0에 수렴시킨 방법 두 가지가 유효했다.

- **문서 순서대로 앞에서부터 일치**시킨다. 건수를 줄이는 게 목표가 아니라 맨 앞 차이를 고치고
  다시 재면 그다음 차이가 드러난다. 건수는 마지막에 0으로 수렴할 뿐 진척 지표가 아니다.
- **위키 UI는 무시가 아니라 제거**: `known-differences`는 차이를 통과시킬 뿐 정렬은 어긋난 채라
  이후가 전부 밀린다. the seed에만 있는 편집 링크(`wiki-edit-section`)·분류 바(`wiki-categories`)처럼
  마크업 산출물이 아닌 것은 정규화가 서브트리째 들어내야 양쪽이 다시 맞물린다(`is_dropped_subtree`).

### 미해결 문법 리포트 (`parity report`)

```
cargo run -p parity -- report                 # 코퍼스 대상
cargo run -p parity -- report <dir>/*.namu    # 임의 문서 뭉치 (덤프 표본 등)
```

마커는 정상 소비했지만 우리가 **조용히 버린** 것을 빈도순으로 집계한다. 미지원 매크로와
무시한 이미지 옵션이 여기 잡힌다. residue는 0인데 화면은 다른 부류라, 이 도구가 없으면
렌더 대조 없이는 놓친다.

해석 시 주의: 매크로 이름 판정이 `char::is_alphanumeric`이라 `[V]`, `[Chorus]` 같은 평범한
대괄호 텍스트가 매크로로 잡힌다([구현 현황](../spec/implementation-status.md)의 파싱 한계 참고).
**빈도가 높은 이름**에 주목한다 —
실제 문서가 널리 쓰는 이름이면 the seed가 지원할 가능성이 크다.

### residue 스캔

```
cargo run -p namumark-parser --example residue_scan -- <문서디렉토리>/*.namu
```

파싱 후에도 일반 텍스트에 남은 마크업 토큰(residue)을 집계한다. 미지원 문법을 찾는
1차 도구다. 골든 스냅샷 헤더의 `residue` 줄도 같은 지표다.

한계 두 가지를 알고 써야 한다.

- **거짓 양성**: 실제 문서에는 `'''`나 `[[`를 닫지 않은 오작성이 흔하다. 이건 나무위키에서도
  깨져 보이므로 우리 버그가 아니다. 문법 도움말 문서는 문법을 *설명*하느라 예제를 리터럴로
  싣기 때문에 특히 신호 대 잡음비가 나쁘다.
- **거짓 음성**: 마커가 소비되기만 하면 residue는 0이다. 파라미터를 잘못 해석하거나 조용히
  무시하는 차이는 잡지 못한다(예: 이미지의 미지원 옵션은 폐기되어 화면에도 residue에도 안 남는다).

그래서 residue 스캔은 **후보 발굴**용이고, 확정은 위의 렌더 대조로 한다.

### 골든 스냅샷

`fixtures/`의 실제 문서 12건에 대한 AST·마크업 골든이 파리티 게이트다. 구조를 바꿔도
골든이 무변경으로 통과해야 한다.

여기에 **문법 회귀 코퍼스**(`fixtures/corpus/`, 287건)가 짝으로 붙는다. 실제 문서 골든은
"문서 전체가 그대로인가"만 지켜서 무엇이 깨졌는지는 알려주지 않는다. 코퍼스는 케이스가
문법 하나로 격리돼 있어 깨진 문법을 파일명이 곧장 가리킨다 — 최악의 케이스(문법 도움말
전체)보다 작은 케이스가 원인 분리에 낫다는 처방이다. 등급 체계와 갱신법은 `fixtures/corpus/README.md`.

코퍼스 케이스는 `.case` 파일 하나가 근거 등급·원문·의미론·렌더링을 모두 담아
**자기완결적**이다(사용자 결정). 파일 하나만 열면 무슨 근거로 이렇게 해석되고 이렇게
그려지는지가 전부 보인다. 골든을 계층별 디렉토리로 흩어 두면 케이스 하나를 이해하는 데
파일 셋을 오가야 하고, 원문과 산출물이 따로 놀 여지가 생긴다.

구획은 **첫 줄이 밝힌 줄 수**로 가른다(사용자 결정). 본문에 구분자를 두지 않는다. 코퍼스
원문은 온갖 마크업이라 어떤 구분자든 원문에 나타날 수 있어 — `===`는 문단, `---`는
수평줄이다 — 구분자 방식은 결국 구분자를 피해 다니는 우회로를 낳는다. 길이로 가르면
충돌이 성립하지 않아 원문을 그대로 싣는다.

코퍼스는 `EmptyContext`가 아니라 **고정된 작은 위키**(`CorpusContext`) 위에서 렌더한다.
`EmptyContext`는 파일도 문서도 시각도 없는 세계라 이미지가 전부 "파일 없음" 링크로,
날짜 매크로와 `[include]`가 원문 표기로 주저앉는다. 그러면 이미지 레이아웃·상대 링크·
include 전개·날짜 계산이 한 번도 실행되지 않은 채 골든이 통과한다(실제로 그랬다 —
`<img>`를 낸 케이스가 0건이었다). `#!if`의 변수 바인딩과 `calleeTitle`은 `expand_include`
안에서만 스코프가 생기므로, 그 경로를 지나려면 케이스가 `[include(틀:...)]`를 거쳐야 한다.

코퍼스 케이스는 근거 등급 없이는 등록되지 않는다(골든 테스트가 실패시킨다). 다만 등급은
`도움말예제`/`도움말서술`/`미확인` 셋이며 **`렌더확정`은 쓰지 않는다** — 아래 표의
`렌더확정`은 렌더 HTML에서 대응 출력을 실제로 확인했다는 뜻인데, 코퍼스는 아직 케이스별로
그 확인을 하지 않았다. `도움말예제`는 그 확인이 가능하다는 뜻이지 했다는 뜻이 아니다.

**임의의 마크업은 실측할 수 없다.** 알파위키에 우리가 지어낸 마크업을 넣어 the seed로
렌더시키는 읽기 전용 경로는 없다 — `/preview`는 POST 전용이고 쓰기 경로는 쓰지 않는다.
그래서 깨진 문법의 the seed 동작은 도움말이 서술하거나 실제 문서가 우연히 그 패턴을
쓰지 않는 한 확정할 수 없고, 코퍼스의 `미확인` 89건이 그 몫이다. 그쪽 골든은 파리티
근거가 아니라 **현 동작 보존** 장치로만 기능한다.

## 성과와 남은 차이

### 파리티 0

문법 도움말·심화 두 문서를 앞에서부터 차례로 일치시켜 차이가 13,571건에서 0으로 수렴했다.
그 과정에서 규명한 나무마크 문법은 [스펙 문서](../spec/namumark.md)와 `fixtures/corpus/`가
정본으로 담는다. 이 문서는 개별 문법이 아니라 그 규명을 가능케 한 **검증 방법**을 남긴다.

### 하네스가 아니면 못 잡는 부류

`residue`도 골든도 못 보는데 `parity compare`만 잡은 대표 사례다 — 하네스를 만든 이유가
여기 있다.

- **표 전체(`table`) 스코프 속성 누락** — `tablewidth`·`tablebgcolor` 등이 화면에서 통째로
  사라졌다. 그런데 마커는 정상 소비되어 `residue`는 0이고, 골든은 우리 출력끼리만 비교하니
  무변경으로 통과했다. the seed 렌더 대조만이 잡았다.
- **`#!wiki` style 화이트리스트** — the seed는 `image-rendering`·무효 값(`display: 5ine`)·
  함수 안 함수가 든 값을 걸러 낸다. 우리는 무필터로 통과시켜 도움말 서술과도 어긋났다.
  렌더로 통과 속성을 목록화해 `style.rs`(`SupportedStyle`)로 재현했다.

### 남은 차이

파리티 0 이후 남은 차이(미구현 매크로·옵션, 버그성 차이, 파싱 한계, 우리가 더 관대한 곳)는
[구현 현황 문서](../spec/implementation-status.md)가 정본으로 유지한다. 파리티 하네스가 새
차이를 발견하면 그쪽에 반영한다.

여기서는 확인된 **일치** 두 가지만 기록해 둔다: 표 옵션 공백형(`<table align=center>`)은
`namumark-text`가 옵션 이름의 공백을 제거해 정규화하므로 자연히 맞고, `##@`의 읽기 모드
미출력도 일치한다.
