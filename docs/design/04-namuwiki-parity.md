# 설계: 나무위키 파리티 검증

상태: 검증 체계 확립 (2026-07)

## 문제

나무위키 본체(the seed)는 비공개라 나무마크의 공식 스펙이 없다. namu.wiki는 크롤링을
차단한다(403). 따라서 "우리 구현이 실제 나무위키와 의미론상 일치하는가"를 직접 확인할
1차 근거가 없다.

## 근거 우선순위

같은 문법이라도 근거마다 신뢰도가 다르다. 판단이 갈리면 위쪽을 따른다.

| 순위 | 근거 | 성격 | 한계 |
|---|---|---|---|
| 1 | **the seed 엔진의 실제 렌더링 결과** | 확정 | 나무위키 본체는 접근 불가 → 아래 대체 경로 |
| 2 | **나무위키 문법 도움말 원문** | 스펙 서술 | 서술과 실제 동작이 어긋날 수 있음 |
| 3 | 실제 문서 덤프의 사용 양상 | 빈도·실재성 | 문서 원문 자체의 오작성이 섞임 |
| 4 | openNAMU 렌더러 소스 | 참고 | 나무위키에 없는 자체 확장이 섞여 있어 단독 근거로 쓰지 않는다 |

4순위를 낮게 두는 이유의 실례: openNAMU에 있는 `{{{#!if}}}`를 "openNAMU 확장"으로 판단한
적이 있는데, 실제로는 **the seed도 지원한다**(`틀:상위 문서`가 쓴다). openNAMU에 있다고
확장인 것도, 없다고 없는 것도 아니다. 판단은 1·2순위 근거로 한다.

### the seed 렌더링 결과를 얻는 경로

**알파위키**(`www.alphawiki.org`)가 나무위키와 **같은 the seed 엔진**을 쓰면서 원문과
렌더링 결과를 모두 공개한다. 나무위키 대신 여기를 1차 근거로 삼는다.

- 렌더링 결과: `GET /w/<문서명>` — HTML 그대로
- 원문 마크업: `GET /raw/<문서명>` — 본문은 `window.INITIAL_STATE`에 base64(zlib(protobuf))로 들어 있다.
  textarea는 비어 있고 클라이언트가 채우므로 HTML만 봐서는 안 되고 이 값을 풀어야 한다.

알파위키의 `알파위키:문법 도움말`, `알파위키:문법 도움말/심화`는 문서 말미의
`틀:문서 가져옴/나무위키`가 밝히듯 **나무위키의 문법 도움말을 그대로 가져온 문서**다.
따라서 이 두 문서는 순위 2(스펙 서술)인 동시에, 그 렌더링 결과가 순위 1(실제 동작)이다.

문법 도움말은 예제를 "입력 / 출력" 표로 나란히 싣는다. **원문에서 예제 마크업을 찾고
렌더 HTML에서 대응 출력을 찾으면 the seed의 실제 동작이 확정된다.** 이것이 이 저장소가
쓰는 대조 기법이다.

읽기(GET)만 사용한다. 편집·미리보기 등 쓰기 경로는 쓰지 않는다.

### 실제 문서 덤프

HuggingFace `heegyu/namuwiki` 덤프(2022-03-01, 867,024건)의 datasets-server rows API로
무작위 표본을 얻어 빈도를 잰다. 발견한 차이가 실제로 얼마나 흔한지 판단하는 데 쓴다.
문서 본문은 CC BY-NC-SA 2.0 KR이므로 `fixtures/`의 라이선스 분리 규정을 따른다.

## 도구

세 도구가 서로 다른 사각지대를 본다. **AI 없이 돌아가는 것이 요구사항이다** — 매번 대조를
사람이나 모델이 하면 비용이 한계를 정한다.

| 도구 | 잡는 것 | 네트워크 |
|---|---|---|
| `parity compare` | the seed 렌더와 우리 렌더의 구조 차이 | 코퍼스 fetch 시에만 |
| `parity report` | 우리가 **조용히 버린** 매크로·옵션 | 불필요 |
| `residue_scan` | 우리가 **텍스트로 흘린** 마크업 | 불필요 |

### 파리티 대조 (`parity compare`)

```
python3 tools/fetch-parity-corpus.py     # 코퍼스 수집 (target/parity-corpus/, 커밋 안 함)
cargo run -p parity -- compare           # the seed ↔ 우리 렌더 대조
```

the seed 렌더 결과와 우리 백엔드 출력을 **구조 지문**으로 정규화해 대조한다. 두 출력은
골격(태그·중첩·style 시맨틱)이 같고 표현만 다르므로, 표현 차이를 걷어내면 남는 diff가
곧 의미론 차이다.

| 표현 차이 | the seed | 우리 | 정규화 |
|---|---|---|---|
| 클래스 어휘 | `class='t-m-VAGP'` | `class="wiki-table"` | 양쪽 제거 |
| 색상 표기 | `rgb(171,205,239)` | `#abcdef` | 6자리 hex로 통일 |
| style 순서·공백 | `width:30%; text-align:center` | `text-align: center; width: 30%` | 파싱 후 이름순 정렬 |
| 다크 색상 | `data-dark-style` | CSS 변수 + `.dark` | 비교 제외 (별도 과제) |

`tools/parity/known-differences.txt`에 등록한 패턴에 걸리는 차이는 통과시키고, **새 차이만**
보고하며 있으면 종료 코드 1이다. 의도된 차이를 등록할 때는 근거를 주석으로 남긴다.

코퍼스 수집(`fetch-parity-corpus.py`)은 `[include(틀:...)]`가 참조하는 틀 문서까지 재귀로
받는다. 파리티 도구가 include를 확장하려면 틀 원문이 필요하기 때문이다. 문서 본문은
CC BY-NC-SA 2.0 KR이라 캐시는 커밋하지 않는다(`target/` 아래).

**건수는 진척도의 지표가 아니다.** 어긋난 지점 하나가 뒤따르는 조각들의 정렬을 통째로
밀어내 건수를 부풀린다. `compare`는 두 가지를 함께 보여준다.

- **케이스마다 맨 앞 차이 3건** — 여기가 근원이다. 뒤따르는 것은 대개 밀려서 나온 결과다.
  구조가 어긋나는 유형(태그가 다른 경우)은 이쪽을 봐야 원인이 잡힌다.
- **차이 유형 분포** — 조각 하나짜리 차이(속성 값 등)는 정렬을 밀지 않으므로 분포 상위에
  그대로 쌓인다. 이쪽은 개수가 곧 영향도다.

작업 순환은 이렇다: `compare` 실행 → 맨 앞 차이로 근원 파악(또는 유형 상위로 영향도 파악)
→ the seed 렌더 HTML에서 실제 동작 확정 → 수정 → 재측정. **여기에 AI가 필요 없다.**

**현재 상태: baseline 정리 전이라 게이트가 아니다.** 문법 도움말 2건 기준 3,381건이
남아 있다(13,571건에서 감소). 남은 근원은 이미지 렌더 구조와 코드블록 주변이다.
baseline을 0에 수렴시킨 뒤 CI 게이트로 전환하는 것이 순서이고, 그때까지는 리포트 도구로 쓴다.

노이즈를 줄이는 두 방향이 남아 있다.

- **케이스를 작게**: 문법 도움말은 온갖 문법에 틀·이미지가 얽힌 최악의 케이스다.
  단순한 문서를 여러 건 모으는 편이 원인 분리에 낫다.
- **위키 UI는 무시가 아니라 제거**: `known-differences`는 차이를 통과시킬 뿐 정렬은 어긋난 채라
  이후가 전부 밀린다. the seed에만 있는 편집 링크(`wiki-edit-section`)처럼 마크업 산출물이
  아닌 것은 정규화가 서브트리째 들어내야 양쪽이 다시 맞물린다(`is_dropped_subtree`).

### 미해결 문법 리포트 (`parity report`)

```
cargo run -p parity -- report                 # 코퍼스 대상
cargo run -p parity -- report <dir>/*.namu    # 임의 문서 뭉치 (덤프 표본 등)
```

마커는 정상 소비했지만 우리가 **조용히 버린** 것을 빈도순으로 집계한다. 미지원 매크로와
무시한 이미지 옵션이 여기 잡힌다. residue는 0인데 화면은 다른 부류라, 이 도구가 없으면
렌더 대조 없이는 놓친다.

해석 시 주의: 매크로 이름 판정이 `char::is_alphanumeric`이라 `[V]`, `[Chorus]` 같은 평범한
대괄호 텍스트가 매크로로 잡힌다(아래 "알려진 차이" 참고). **빈도가 높은 이름**에 주목한다 —
실제 문서가 널리 쓰는 이름이면 the seed가 지원할 가능성이 크다.

### residue 스캔

```
cargo run -p namumark-parser --example residue_scan -- <문서디렉토리>/*.namu
```

파싱 후에도 일반 텍스트에 남은 마크업 토큰(residue)을 집계한다. 미지원 문법을 찾는
1차 도구다. 골든 스냅샷 헤더의 `residue` 줄도 같은 지표다.

한계 두 가지를 알고 써야 한다.

- **거짓 양성**: 실제 문서에는 `'''`나 `[[`를 닫지 않은 오작성이 흔하다. 이건 나무위키에서도
  깨져 보이므로 우리 버그가 아니다. 문법 도움말 문서는 문법을 *설명*하느라 예제를 리터럴로
  싣기 때문에 특히 신호 대 잡음비가 나쁘다.
- **거짓 음성**: 마커가 소비되기만 하면 residue는 0이다. 파라미터를 잘못 해석하거나 조용히
  무시하는 차이는 잡지 못한다(예: 이미지의 미지원 옵션은 폐기되어 화면에도 residue에도 안 남는다).

그래서 residue 스캔은 **후보 발굴**용이고, 확정은 위의 렌더 대조로 한다.

### 골든 스냅샷

`fixtures/`의 실제 문서 12건에 대한 AST·마크업 골든이 파리티 게이트다. 구조를 바꿔도
골든이 무변경으로 통과해야 한다.

여기에 **문법 회귀 코퍼스**(`fixtures/corpus/`, 287건)가 짝으로 붙는다. 실제 문서 골든은
"문서 전체가 그대로인가"만 지켜서 무엇이 깨졌는지는 알려주지 않는다. 코퍼스는 케이스가
문법 하나로 격리돼 있어 깨진 문법을 파일명이 곧장 가리킨다. 위 "케이스를 작게"가 노이즈를
줄이는 방향이라 한 것과 같은 처방이다. 등급 체계와 갱신법은 `fixtures/corpus/README.md`.

코퍼스 케이스는 `.case` 파일 하나가 근거 등급·원문·의미론·렌더링을 모두 담아
**자기완결적**이다(사용자 결정). 파일 하나만 열면 무슨 근거로 이렇게 해석되고 이렇게
그려지는지가 전부 보인다. 골든을 계층별 디렉토리로 흩어 두면 케이스 하나를 이해하는 데
파일 셋을 오가야 하고, 원문과 산출물이 따로 놀 여지가 생긴다.

구획은 **첫 줄이 밝힌 줄 수**로 가른다(사용자 결정). 본문에 구분자를 두지 않는다. 코퍼스
원문은 온갖 마크업이라 어떤 구분자든 원문에 나타날 수 있어 — `===`는 문단, `---`는
수평줄이다 — 구분자 방식은 결국 구분자를 피해 다니는 우회로를 낳는다. 길이로 가르면
충돌이 성립하지 않아 원문을 그대로 싣는다.

코퍼스는 `EmptyContext`가 아니라 **고정된 작은 위키**(`CorpusContext`) 위에서 렌더한다.
`EmptyContext`는 파일도 문서도 시각도 없는 세계라 이미지가 전부 "파일 없음" 링크로,
날짜 매크로와 `[include]`가 원문 표기로 주저앉는다. 그러면 이미지 레이아웃·상대 링크·
include 전개·날짜 계산이 한 번도 실행되지 않은 채 골든이 통과한다(실제로 그랬다 —
`<img>`를 낸 케이스가 0건이었다). `#!if`의 변수 바인딩과 `calleeTitle`은 `expand_include`
안에서만 스코프가 생기므로, 그 경로를 지나려면 케이스가 `[include(틀:...)]`를 거쳐야 한다.

코퍼스 케이스는 근거 등급 없이는 등록되지 않는다(골든 테스트가 실패시킨다). 다만 등급은
`도움말예제`/`도움말서술`/`미확인` 셋이며 **`렌더확정`은 쓰지 않는다** — 아래 표의
`렌더확정`은 렌더 HTML에서 대응 출력을 실제로 확인했다는 뜻인데, 코퍼스는 아직 케이스별로
그 확인을 하지 않았다. `도움말예제`는 그 확인이 가능하다는 뜻이지 했다는 뜻이 아니다.

**임의의 마크업은 실측할 수 없다.** 알파위키에 우리가 지어낸 마크업을 넣어 the seed로
렌더시키는 읽기 전용 경로는 없다 — `/preview`는 POST 전용이고 쓰기 경로는 쓰지 않는다.
그래서 깨진 문법의 the seed 동작은 도움말이 서술하거나 실제 문서가 우연히 그 패턴을
쓰지 않는 한 확정할 수 없고, 코퍼스의 `미확인` 77건이 그 몫이다. 그쪽 골든은 파리티
근거가 아니라 **현 동작 보존** 장치로만 기능한다.

## 확인된 차이

### 수정 완료

아래는 모두 `parity compare`가 찾아낸 것이다. 문법 도움말 2건 기준 차이가 13,571건에서
2,588건으로 줄었다(81% 감소).

**`{{{#!if 조건식}}}`** (2026-07 구현)

나무위키의 실제 문법이고 `틀:상위 문서`·`틀:하위 문서`·`틀:상세 내용` 등이 쓴다.
단순한 조건이 아니라 **값을 내면서 변수도 만드는** 작은 표현식 언어다.

```
{{{#!if top = 문서명1 != null ? 문서명1 : calleeTitle
상위 문서: [[@top@]]}}}
```

조건식이 만든 `top`을 내용의 `@top@`이 참조한다. 그래서 평가는 참/거짓과 변수 바인딩을
함께 돌려준다(`condition.rs`). 문법 범위는 나무위키 틀에서 실제로 관찰된 만큼이다 —
시퀀스(`,`), 대입, 삼항, `||`, **`&`**(나무위키 틀은 논리 AND에 단일 앰퍼샌드를 쓴다),
`==`/`!=`, `+`, `.length`, `startsWith`/`substr`/`lastIndexOf`, `?.`, 괄호, `null`.
`calleeTitle`(호출자 문서명)은 `WikiContext::current_title()`이 공급한다.

처리 순서가 중요하다. 조건식이 만든 변수를 본문의 `@이름@`이 참조하므로 **인자를 치환하기
전에** `#!if`를 먼저 해결해야 한다(`resolve.rs`의 `apply_conditionals`). 변수는 뒤따르는
`#!if`로 누적된다 — `틀:하위 문서`가 `c`·`l`을 앞에서 만들고 뒤에서 재사용한다.

**픽셀 값의 소수** (2026-07 수정) — 나무위키는 버린다(`width=21.6px` → `21px`).

**`{{{#!html}}}` sanitizer** (2026-07 도입)

나무위키는 `#!html`을 원시 HTML로 렌더한다 — `{{{#!html <span style="…">글</span>}}}`이
`<span style="background-color:#999">글</span>`이 되고 `&nbsp;`는 U+00A0이 된다(렌더확정).
우리는 그때까지 이스케이프된 코드 박스로 내보내고 있었다.

나무마크는 위키 사용자가 쓰는 입력이고 `#!html`은 그 입력이 원시 HTML로 나가는 유일한
통로다. 그래서 **화이트리스트**다 — 아는 것만 통과시킨다. 나무위키가 정확히 무엇까지
허용하는지는 문서에 위험한 예제가 없어 렌더로 확정할 수 없고, 도움말도 `#!html`을
"비권장, 지원 종료 가능"이라 서술한다. 그래서 허용 범위는 도움말이 실제로 보여주는
표현으로 좁혔다.

- 태그: 텍스트 서식(`b`·`i`·`u`·`s`·`strong`·`em`·`sub`·`sup`·`small`·`big`·`code`)과
  `br`·`wbr`·`span`·`div`까지. 스크립트·삽입·폼·미디어는 없다.
- 속성: `class`·`style`만. `id`는 문서 앵커(`s-1`, `fn-1`)와 충돌할 수 있어 받지 않고,
  `href`·`src`는 나무마크의 링크·이미지 문법이 대신하므로 열지 않는다.
- `style`은 `url(`·`expression(`·`javascript:`·`@import`·`behavior:`가 있으면 **통째로**
  버린다. 부분만 지우면 남은 값이 무엇이 될지 장담할 수 없다.
- 허용하지 않는 태그는 껍데기만 벗기고 글은 남긴다. 단 `script`·`style`·`iframe`·
  `object`·`embed`는 **내용 자체가 코드**라 내용까지 버린다.
- 열어 둔 태그는 sanitizer가 닫는다 — 새어 나가면 바깥 문서 구조가 무너진다.

**표 셀의 문단 래퍼** (2026-07 수정)

나무위키는 `<td>` 안에 내용이 무엇이든 **항상** `<div class='wiki-paragraph'>`를 둔다
(문법 도움말의 td 990개가 예외 없이 그렇다). 우리는 셀 내용이 블록이면 이 래퍼를 생략해
구조가 어긋났고, 그 지점부터 정렬이 밀려 차이가 대량으로 불어났다.

**각주 참조 구조** (2026-07 수정)

```html
<a class='wiki-fn-content' title='<각주 내용 글자>' href='#fn-1'><span id='rfn-1'></span>[1]</a>
```

각주 내용의 글자를 `title`에 실어 툴팁으로 보여주고, 본문 복귀 앵커(`#rfn-N`)는 링크 안쪽
빈 span이 갖는다. 우리는 툴팁이 없고 앵커를 `<a>`에 달고 있었다. IR의
`FootnoteReference`에 `tooltip`을 더했다(각주 라벨과 마찬가지로 자기완결).

**중첩 include(틀 속의 틀) 확장** (2026-07 수정) — 가장 큰 원인이었다.

나무위키는 틀 속의 틀을 확장하지 않는다(문법 도움말이 "include 중첩 불가"라고 명시).
원문을 노출하지도 않고 **조용히 버린다**. 우리는 깊이 8까지 재귀 확장하고 있었다.

틀 문서 끝에는 보통 `[include(틀:X/설명 문서)]`가 붙어 있다. 우리는 그 틀을 쓴 문서에
설명 문서의 헤딩까지 딸려오게 만들어, 심화 문서 목차가 39개여야 할 것이 54개가 됐다.

이 규칙 덕에 순환 include도 구조적으로 성립하지 않아, 깊이 제한과 순환 스택 검사가
함께 필요 없어졌다(`include_stack: Vec<String>` → `inside_include: bool`).

**목차 제목 평탄화** (2026-07 수정)

나무위키 목차는 제목의 서식·링크·앵커를 버리고 **글자만** 싣는다
(`== [[/TeX|수식]] ==` → `수식`, `=== [anchor(include)]틀·문서 삽입 ===` → `틀·문서 삽입`).
본문 헤딩에는 앵커가 그대로 남는다 — 목차에서만 걷어낸다.

이에 맞춰 IR도 `TableOfContentsEntry::title`을 `Vec<RenderInline>`에서 `String`으로 바꿨다.
목차 제목이 글자뿐임을 타입이 드러낸다.

**외부 링크 속성** (2026-07 수정)

`rel='nofollow noopener ugc'`(우리는 `ugc` 누락에 순서도 달랐다)와 `title=<URL>`.

**표 전체(`table`) 스코프 속성 누락** (2026-07 수정, `parity compare`가 발견)

`tablewidth`·`tablebgcolor`·`tablealign`·`tablebordercolor`가 화면에서 통째로 사라졌다.
AST·IR에는 `TableAttributeScope::Table`로 보존되는데 백엔드가 `Row`·`Cell`·`Column`
스코프만 방출하고 있었다. 나무위키의 거의 모든 틀이 쓰는 옵션이라 실제 문서 10건의
골든이 바뀌었다.

residue는 0이고(마커는 정상 소비된다) 골든은 우리 출력끼리만 비교하니 무변경으로
통과한다 — **렌더 대조만이 잡을 수 있는 부류**이고, 하네스를 만든 이유다.

나무위키는 표 스코프 속성을 두 군데로 나눠 싣는다. 정렬은 감싸는 div의 클래스로,
너비는 div의 style로(이때 표 자신은 `width:100%`), 색·테두리는 table의 style로 간다.

```html
<div class='wiki-table-wrap table-center' style='width:500px'>
  <table class='wiki-table' style='background-color:#fff; width:100%; border:2px solid #ff0000;'>
```

**문서 경로의 `:`·`/` 인코딩** (2026-07 수정)

나무위키는 이름공간 구분자 `:`와 하위 문서 구분자 `/`를 인코딩하지 않는다
(`/w/분류:파일/링크 아이콘` → `/w/%EB%B6%84%EB%A5%98:%ED%8C%8C%EC%9D%BC/...`).
우리는 unreserved 문자만 통과시켜 `%3A`로 인코딩하고 있었다.

**목차 구조** (2026-07 수정)

나무위키 목차는 `<details open><summary></summary>`로 감싸 접을 수 있고, 깊이를
`<div class='toc-indent'>` **중첩**으로 표현하며, 항목은 `<span class='toc-item'>`이고
번호 뒤의 점은 **링크 밖**에 있다(`<a href='#s-1'>1</a>. 개요`).

우리는 `toc-head` + `toc-item toc-indent-N` 평면 구조에 점이 링크 안이었다. 헤딩(`<h2>`)은
원래부터 일치했다 — 점이 링크 안(`<a id='s-1' href='#toc'>1.</a>`)인 것이 맞다.

**줄머리 문맥 안에서 열린 여러 줄 `{{{` 그룹** (2026-07 수정)

인용문·리스트·들여쓰기 안에서 `{{{` 그룹이 열리고 마커 없는 다음 줄로 이어지면, 우리는
그룹을 미완결로 보고 원문 텍스트로 흘렸다. the seed는 그룹이 닫힐 때까지를 그 문맥의
내용으로 본다.

```
>{{{#!wiki style="margin:1em calc(1em + 25px) 1em 1em"
인용문}}}
```

the seed 렌더 결과(문법 도움말 "인용문" 문단):

```html
<blockquote><div><div style='margin:1em calc(1em + 25px) 1em 1em'>인용문</div></div></blockquote>
```

리스트도 같다(심화 문서 "틀 테두리 색" 문단의 배지 예제가 `<li><div style='display:inline;background:orange;…'>`로 렌더된다).
표 셀은 `is_row_complete`가 같은 상황을 이미 처리하고 있어 영향이 없었다.

원인은 줄머리 마커가 붙은 줄만 모아 하위 영역을 만든 것이다. `block.rs`의
`collect_marked_lines`(인용·들여쓰기)와 `closing_line_of_open_group`(리스트 항목)이
열린 그룹을 이어받고, 끝까지 닫히지 않으면 마커 규칙만으로 되돌아간다.

회귀 테스트는 `nesting_tests.rs`의 `*_containing_multiline_*` 5건.

표본 875건 중 이 패턴을 쓰는 문서는 1건(약 0.1%)으로 빈도는 낮지만, 나무위키의 둘러보기
틀·접기에서 쓰는 관용이라 롱테일에서 계속 나타난다.

### 미수정 차이

증거 등급을 함께 적는다. **렌더확정**은 the seed 렌더 HTML 조각을 확보한 것이고,
**서술만**은 도움말 서술만 있고 실사용 예가 없어 렌더로 확정하지 못한 것이다.
서술만인 항목은 도움말 서술과 실제 동작이 어긋날 수 있으니, 구현 전 렌더로 재확인한다.

| 항목 | the seed | 우리 동작 | 증거 |
|---|---|---|---|
| 문단명 앵커 | 헤딩 제목을 `<span id='개요'>`로 감싸 `[[#개요]]`용 앵커 생성 | `s-N` 앵커만 방출 | 렌더확정 |
| 표 셀의 여러 문단 | 셀 내용이 여러 줄이어도 `wiki-paragraph` **하나** 안에 담는다 | 문단마다 `wiki-paragraph`를 만들어 중첩된다 | 렌더확정 |
| 셀 정렬의 명시/암묵 구분 | 공백 없는 기본 왼쪽(`\|\|텍스트\|\|`)은 style을 생략하고, 뒤 공백으로 명시한 왼쪽(`\|\|텍스트 \|\|`)은 `text-align:left`를 방출한다 | `HorizontalAlignment::Left`에 이 구분이 없어 항상 방출한다. **시각적 결과는 같다** — 마크업까지 맞추려면 AST 변경이 필요하다 | 렌더확정 |
| `[[분류:X#blur]]` | href에서 `#blur` 제거 + 분류바 `<li>`에 클래스 | **분류명이 `"X#blur"`로 오염** (`lower.rs`가 앵커를 분리하지 않음) | 렌더확정 |
| `[[../]]`, `[[/하위]]` | 상위/하위 문서로 해석 (상위 없으면 자기 문서 + 볼드) | target을 문자열 그대로 보존 | 렌더확정 |
| `[pagecount]`, `[pagecount(이름공간)]` | 정수 출력 (비정식 이름공간이면 전체 수) | 미해결 매크로 → 원문 노출 | 렌더확정 |
| `[vimeo(...)]`, `[navertv(...)]` | 지원 (`player.vimeo.com`, `tv.naver.com/embed` iframe) | 미해결 매크로 → 원문 노출 | 렌더확정 |
| 이미지 `border-radius=` | `style='border-radius: 15px'` (단위 없으면 px) | 조용히 폐기 | 렌더확정 |
| `{{{#!folding}}}` 태그 구조 | `<details><summary>` | `<dl class="wiki-folding"><dt>` — 안내 문구 기본값 `More`는 일치 | 렌더확정 |
| 이미지 `align=middle` (`center` 동의어) | 가운데 정렬 | 정렬 없음으로 폐기 | 서술만 |
| 이미지 `rendering=pixelated` | 지원 | 조용히 폐기 | 서술만 |
| `[youtube(id, start=, end=)]` | `start`/`end` 지원 (YouTube 전용) | 폐기 (`width`/`height`만) | 서술만 |
| `[목차(hide)]` | 접힌 목차 | 인자 무시 → 항상 펼침 | 서술만 |
| `<color#RRGGBB>` (등호 생략형) | 지원 | 옵션 파싱 중단 → 셀 본문 텍스트 | 서술만 |
| `[[\|출력]]` (문서부가 빈 파이프 링크) | 현재 문서 링크 + 볼드 | `target: ""`로만 처리 | 서술만 |
| 분류 `[[분류:X\|정렬키]]` | 분류 문서 색인 정렬키 (표시엔 영향 없음) | 정렬키 폐기 | 서술만 |

표 옵션 공백형(`<table align=center>`)과 `##@`의 읽기 모드 미출력은 **일치**를 확인했다.
전자는 `namumark-text`가 옵션 이름의 공백을 제거해 정규화하므로 자연히 맞는다.

### 우리가 더 관대한 곳

the seed가 거부하는 것을 우리가 받아들이면, 나무위키에서 깨져 보일 원문이 우리에게서만
멀쩡히 렌더된다. 파리티 관점에서는 이것도 차이다.

- `<col bgcolor=>`, `<row bgcolor=>` — 도움말이 미지원이라 명시하는데, 옵션 이름의 공백을
  제거하는 정규화가 `colbgcolor`/`rowbgcolor`로 받아버린다. (공백형이 허용되는 건 `table` 계열뿐이다.)
- `{{{#transparent}}}`, `{{{#RRGGBBAA}}}` — 도움말이 미지원이라 명시하는데, 색상 파싱이
  3/6자리 hex가 아니면 이름 색상으로 통과시켜 유효 CSS로 출력된다.

### 검증 과제

the seed는 `#!wiki`의 `style` 속성을 **화이트리스트로 필터링**한다.
`style="margin: -5px -10px; image-rendering: high-quality;"`가 실제로는
`<div style='margin:-5px -10px'>`로 렌더되어 `image-rendering`이 사라진다(심화 도움말은
이 문법이 동작하는 것처럼 서술하지만 렌더는 무시한다). 우리는 `style`을 무필터로 통과시킨다.
어떤 속성이 통과되는지 목록화가 필요하다.

### 알려진 차이 (수정 안 함)

- 매크로 이름 판정이 `char::is_alphanumeric`이라 `[1]`, `[MacBook]`, `[mɪˈrɑːʒ]` 같은
  평범한 대괄호 텍스트가 매크로로 파싱된다. 미해결 매크로는 원문을 그대로 재구성해
  방출하므로 **최종 화면은 the seed와 같다**. AST 의미만 다르다.
- 서식(`'''` 등)은 줄 단위라 여러 줄 색상 블록을 감싼 서식은 잔여 텍스트로 남는다.
